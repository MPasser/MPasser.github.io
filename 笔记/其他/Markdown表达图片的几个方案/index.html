Markdown表达图片的几个方案

发表于 2023-02-07 分类于 Markdown

前言

作为纯文本文件，Markdown的通用性很强，但代价就是没法像Word那样让图片变成文档的一部分。图片只能游离于文档之外进行管理。

不过Markdown的各种渲染器发展到现在，能展现的效果已经很丰富了，总是有一些替代方案。

在此记录一下本人使用过的几种方案。

PlantUML

PlantUML 能用纯本文生成UML图表。例如文本如下：

优点

  • 纯文本。我觉得是很大的优点。
  • 支持的图表类型很多，甚至mindmap这种非UML图表。

缺点

  • 每种图表基本上都有自己的一套语法，有一定的学习成本。
  • PlantUML类似于插件性质，可能并不是所有渲染器都能支持安装。甚至有点破坏Markdown最初的设计理念。
  • 毕竟不是真图片，总有一些内容是UML图表无法表达的。

ASCII字符画

这里不是指图片转字符画，而是类似ASCIIFlow这种纯手工画的。足够灵活，适合画一些简单的示意图，放在代码块里显示。类似于

优点

  • 通用性强，只是ASCII字符，无论放在哪里，只要是等宽字体，都能很好地显示。
  • ASCII字符画有种特别的美感。

缺点

  • 太麻烦了，真就拿鼠标画画，而且ASCIIFlow页面上的许多操作方式设计仍有提升空间。
  • 在某些地方，如果遭遇了意外换行，整幅图都不能看了。
  • 毕竟不是真图片，总有一些内容是画功没法表达的。

github图床

图床直接使用Github Repository存储，也方便管理。

Github图片地址格式一般是 https://raw.githubusercontent.com/${userName}/${repoName}/${branchName}/${path}/${imgFilename}

国内一般无法直接访问，需要用CDN加速，这里用的是staticaly

直接替换域名即可，将raw.githubusercontent.com替换为cdn.staticaly.com/gh

例如原图的格式为 https://raw.githubusercontent.com/MPasser/ImgStorage/master/NoteBackup/笔记/Unity/screenshots2002-12-02.png 替换成 https://cdn.staticaly.com/gh/MPasser/ImgStorage/master/NoteBackup/笔记/Unity/screenshots2002-12-02.png 即可。

当然，要是有长期的域名和服务器，自己搭个图床更好。

[20231008更新] cdn.staticaly.com 果然挂掉了，换用 cdn.jsdelivr.net。CDN格式为https://cdn.jsdelivr.net/gh/{userName}/{repoName}@{branchName}/{path}/{imgFilename}

cdn节点的数据更新有延迟，可手动刷新，将图片链接中的cdn.jsdelivr.net替换为purge.jsdelivr.net之后进行访问，返回的JSON中status为finished即为刷新成功。

还尝试过PicGo+各种图床的方案。没使用的原因是这些图床都不能创建文件夹管理已上传的图片，PicGo也不提供这种功能，上传的所有图片就全部摊在根目录，不太喜欢这种方式。

优点

  • 显示的是真正的图片，限制更少了。
  • 图片的文件结构能够和文档结构相对应，便于管理。

缺点

  • 图片要游离于文档之外管理，就觉得没纯文本编辑那么舒服了。
  • gitee原来也能作图床的，但后来官方加了防盗链，导致所有外部引用图片失效。github在未来也不是没可能加防盗链。真这样的话就太惨了。但至少图片文件结构还在。
  • 除了依赖github外，还需要依赖CDN，这个在未来也不是没可能失效。不过运气好的话还能找到其他的CDN，届时项目内全局替换CDN地址就可以了。

本地引用

试过一回，不作考虑。

图片设置相对路径，随便换个环境就失效了。
