线程基础、线程之间的共享和协作

发表于 2022-05-05 分类于 多线程

基础概念

CPU核心数和线程数的关系

CPU核心数 : 线程数 = 1:1

使用英特尔研发的超线程技术之后，CPU核心数 : 线程数可以变成 1:2

CPU时间片和轮转机制

CPU轮转机制也就是RR调度，Round-Robin，轮询调度

上下文切换，也就是时间片切换的时候，需要保存原先运行进程的状态，加载即将运行进程的状态，大约会花费5000-20000个时钟周期，这个消耗并不少。

操作系统会按照自身的统计值来决定在每个进程上花费时间片的大小，程序无法控制。

什么是进程和线程

  • 进程：程序运行资源分配的最小单位，进程内部有多个线程，会共享这个进程的资源。
  • 线程：CPU调度的最小单位。

澄清并行和并发

  • 并行：同一时刻，可以处理事情的能力。
  • 并发：与时间单位相关，某个时间段内可以同时处理的事情。

高并发编程的意义、好处和注意事项

好处：

  • 充分利用CPU资源
  • 充分缩短用户请求的响应时间
  • 使代码模块化、异步化

注意事项：

  • 多线程之间会共享资源，可能会存在冲突
  • 线程数越多，对硬件配置要求也就越高

认识Java里的线程

Java里的程序天生就是多线程的。

启动新线程的方式

  • 继承Thread类。
  • 实现Runnable接口
  • 实现Callable接口

由于Java是单继承，继承Thread类之后，就不能继承其他类，所以一般都使用实现接口的方式。

Runnable和Callable的区别

使用方式差不多，但Callable接口有返回值，Runnable接口没有返回值。

如何安全停止线程

  1. 等待线程自然执行完毕
  2. 线程抛出异常
  3. stop()、resume()、suspend()
  4. interrupt()、interrupted()方法

良好的程序设计中，Java线程间应是协作式的，而不是抢占式的。

stop()、suspend()方法会强行停止线程，但不会释放资源。已经被弃用。

interrupt()用于“中断”一个线程，但并非强行停止线程，而是对线程发出停止信号，将中断标志位置为true。
isInterrupted()判断当前线程是否处于中断状态。
static方法interrupted()判断当前线程是否处于中断状态，并将终端标志位改为false。
isInterrupted()和 interrupt()一般会配合使用，先判断是否做了终止标记，如果没有则调用interrupt()方法。终止线程时候先判断isInterrupted()是否有终止标记，如果有，则执行自定义的线程中止逻辑。

如果某方法会抛出InterruptedException时，中断标志位会被置为false；需要在catch语句块中自行调用interrupt()方法

Java线程的生命周期

yield()：将线程从运行状态转到就绪状态。不释放锁。调用这个方法的线程可能立即又抢夺到CPU执行权。

start()方法和run()方法

启动一个线程需要调用对象的start()方法。直接调用run()的话，只是普通的方法调用。

线程的优先级

可设置为1-10，默认为5。

线程的优先级设置后，优先关系在不同的运行环境下是可能有差别的，优先关系并不能得到严格保证。

守护线程

  • 当所有非守护线程结束时，程序即中止，同时杀死所有守护线程。
  • thread.setDaemon()方法将线程设置为守护线程，需要在线程启动前设置，否则会产生异常。
  • 守护线程中的try-finally语句块，finally中的语句此时无法保证一定能执行。

线程间共享

synchronized关键字

同一时刻，相同的synchronized方法或语句块，只能有一处执行

synchronized锁的是对象。
synchronized修饰实例方法时，或synchronize(this)，锁的实例对象。
synchronized修饰静态方法时，锁的是类，即类的class对象

volatile关键字

最轻量的同步机制，但并非线程安全。只能保证可见性，无法保证原子性。

在使用volatile修饰的变量时，一定会重新从主线程读取变量的值，放弃线程中暂存的变量值。

最常用的使用场景，只有一个线程写，多个线程读。

ThreadLocal

线程变量，意思是ThreadLocal中填充的变量属于当前线程，该变量对其他线程而言是隔离的。每个使用该变量的线程都会初始化一个完全独立的实例副本。用于线程间的数据隔离。

常用于数据连接池。

底层利用Map实现，将线程id作为Key。

由于每个线程都会有一个独立的实例副本，所以使用ThreadLocal包装的变量，最好只是简单的基本类型，否则占用的空间会很大。

线程间协作

轮询：难以保证及时性，且系统开销大。

等待和通知

wait() 和 notify()/notifyAll() 方法

这些是Object的对象方法

  • wait()
    作用是使当前线程从运行状态(Running)转为等待，进入等待队列，直到收到notify通知后，才进入锁池队列，等待获取同步锁，回到可运行状态(Runnnable)。
    方法可以有一个long类型的入参，表示等待时间，经过等待时间后仍然没有收到通知的话，自动从等待队列转到锁池队列。
    wait语句会释放锁。
  • notify()
    从等待队列中通知一个持有相同锁的线程，锁不同信号会被忽略。
    notify 唤醒线程的规则是随机唤醒还是顺序唤醒取决于 JVM 的具体实现，作为主流的 HotSpot 虚拟机中的 notify 的唤醒规则是顺序的，也就是 notify 会按照线程的休眠顺序，依次唤醒线程。谁先wait，谁先notify。
  • notifyAll()
    通知等待队列中所有持有相同锁的线程
  • notify()/notifyAll()方法通知是延迟通知，必须等待当前线程执行完所有的同步方法/代码块中的语句退出释放锁才通知wait线程。

这些方法都必须在获得锁的情况下才能调用，否则会出现非法监视状态异常。

等待和通知的标准范式

等待方：

  1. 获取对象的锁
  2. 循环里判断条件是否满足，不满足则调用wait()方法
  3. 条件满足则执行业务逻辑

问：这里的条件，具体指的是什么条件？

答：指的是实际业务运行需要满足的条件。

通知方：

  1. 获取对象的锁
  2. 改变条件
  3. 通知所有在等待对象的方法

应尽量使用notifyAll()，使用notify()可能发生信号丢失的情况。

相反，对于之后提及的Condition类的await()/signal()/signalAll()方法，则应尽量使用signal()方法。

因为wait()是Object的方法，所有线程都能调用，并不指明在等待什么，通知的时候要通知所有；而await()调用时就确定了正在等待的Condition。

在标准等待通知范式下，加上等待超时的方式。

问：为什么不直接 对象.wait(T) ？

答：注意这里循环的条件有两个。如果直接 wait(T) ，但 条件 一直不满足的话，则会重复调用 wait(T) 而无限期等待，设置超时时间也就没意义了。

join()方法

线程A，执行了线程B的join方法，线程A必须要等待线程B执行完成了之后，线程A才能继续执行。

就像C语言的中断一样。

各种方法对锁的影响

  • 线程执行yield()、sleep()方法之后，持有的锁不释放。
  • 调用wait()之前必须要持有锁，在调用wait()时，系统会自动释放这个锁。wait()方法调用结束后，线程会重新获得锁。
  • 调用notify()方法前必须要持有锁，调用notify()方法本身不会释放锁。
  • notify()/notifyAll()方法通知是延迟通知，必须等待当前线程执行完所有的同步方法/代码块中的语句退出释放锁才通知wait线程。
