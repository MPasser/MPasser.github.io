线程池

发表于 2022-06-01 更新于 2023-08-17 分类于 Java

概述

线程的容器。用于：

  • 降低线程创建和销毁的资源消耗；
  • 提高响应速度；
  • 提高线程的可管理性。

尝试实现一个自定义的线程池

有几个缺点：

  • 线程池实例化时，工作线程是预加载的，可能会浪费资源。
  • 当在阻塞队列满，无法设置处理机制，外界线程提交任务时会被阻塞。

线程池的创建

ThreadPoolExecutor，JDK所有线程池实现的父类。

各个参数含义

参数类型 参数名 含义
int corePoolSize 核心线程数。长期存活的线程数。提交任务时，如果已经有corePoolSize个线程在运行，则该任务会保存在阻塞队列中；否则直接创建新线程
int maximumPoolSize 允许的最大线程数
long keepAliveTime 空闲线程存活时间，当非核心线程没有任务时，超过存活时间会被销毁
TimeUnit unit keepAliveTime的单位
BlockingQueue<Runnable> workQueue 保存任务的阻塞队列
ThreadFactory threadFactory 线程池创建线程时调用的工厂方法。通过此方法可以设置线程的优先级、线程命名规则以及线程类型(用户线程还是守护线程)等。
RejectedExecutionHandler handler 拒绝策略：当线程池的任务超出线程池队列可以存储的最大值之后，执行的策略

拒绝策略有以下4种：

  • AbortPolicy：直接抛出异常，默认；
  • CallerRunsPolicy：用调用者所在线程来执行任务；
  • DiscardOldestPolicy：丢弃阻塞队列里最老的任务，即队列头的任务；
  • DiscardPolicy：当前任务直接丢弃。

或者实现一个自定义的 RejectedExecutionHandler去 使用。

提交任务

void execute(Runnable command) 不需要返回值；
Future<T> submit(Callable<T> task) 需要返回值。

问：为什么使用阻塞队列而不是普通队列？

答：一般队列只是有限长度的缓冲区，一旦超出缓冲长度，任务就无法保留。阻塞队列通过阻塞可以保留住当前想要继续入队的任务。阻塞队列带有阻塞和唤醒的功能，不需要额外处理，无任务执行时，线程池利用阻塞队列的take方法挂起，从而维持核心线程的存活，不至于一直占用cpu资源。

问：为什么先添加队列而不是先创建最大线程？

答：在创建新线程的时候，是要获取全局锁的，这时候其他线程会被阻塞，影响整体效率。这样设计的代价低。

关闭线程池

使用方法 shutdown() 或 shutdownNow()

shutdown() 会设置线程池的状态，只会中断没有执行任务的线程。

shutdownNow() 不仅会设置线程池的状态，还会尝试停止正在运行或暂停任务的线程。

合理配置线程池

任务性质大致有几种：计算（CPU）密集型、IO密集型、混合型。

  • 计算密集型：加密、大数分解、正则等。最大线程数适当小一些，推荐设置为机器CPU+1 。
  • IO密集型：读取文件、数据库连接、网络通讯等。最大线程数适当大一些，推荐设置为机器的CPU*2 。
  • 混合型：尽量拆分。IO密集型>>计算密集型，拆分意义不大；IO密集型≈计算密集型，是和拆分。

阻塞队列的选择，因尽量使用有界，无界队列可能会导致内存溢出。

预定义的线程池

由java.util.concurrent.Executors，使用静态方法的方式提供。

FixedThreadPool

创建固定线程数量的，适用于负载较重的服务器，使用无界队列。

SingleThreadPool

创建单个线程，需要顺序保证执行任务，不会有多个线程活动，使用无界队列。

CachedThreadPool

会根据需要来创建新线程，执行很多短期异步任务的程序，使用了SynchronousQueue

WorkStealingPool

基于ForkJoinPool实现

ScheduledThreadPoolExecutor

适合需定期执行的周期性任务

几个比较重要的方法：

  • schedule， 执行一次任务，可以设置延时执行
  • scheduleAtFixedRate，提交固定时间间隔执行的任务
  • scheduleWithFixedDelay，提交固定延时间隔执行的任务

问：scheduleAtFixedRate 和 scheduleWithFixedDelay 有什么区别？

答：scheduleAtFixedRate 中的时间间隔，是每次任务开始时刻的间隔，但是，如果上个任务超时未结束，则下个任务需要等待上个任务执行完毕后立马执行；scheduleWithFixedDelay中的时间间隔，是上一次任务结束，到下一次任务开始时刻之间的间隔。

如果在任务执行中抛出了异常而自身未处理，则整个周期任务会停止，只能重启恢复。

CompletionService

思考如下场景：线程池接收Callable任务，需要有一个容器接收任务的返回值。如果使用阻塞队列存放任务、获取任务返回值的话，由于队列先入先出的性质，处于队列后的任务，总是需要等待处于队列前的任务执行完成才能给出结果，效率会非常低。

即有些任务因为执行时间较长，而在他后面的任务可能执行任务的时间较短，已经提前完成了，但是并不能得到及时的处理。
