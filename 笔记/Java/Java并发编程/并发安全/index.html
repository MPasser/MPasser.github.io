并发安全

发表于 2022-06-04 分类于 多线程

线程安全

在拥有共享数据的多条线程并行执行的程序中，线程安全的代码会通过同步机制保证各个线程都可以正常且正确的执行，不会出现数据污染等意外情况。

线程安全的类一般都要满足 操作的原子性 和 内存的可见性。

线程不安全一般出现在多个线程之间共享状态时。

实现并发安全的方案

栈封闭

方法的局部变量是栈封闭的，多线程访问同一方法，使用到其中的局部变量时，每个线程都有自己的局部变量，不会出现线程安全问题。

无同步方案

无状态

一个类如果没有成员变量，则不涉及共享状态，无需同步措施。

状态不可变

一个类如果所有成员变量都不可变，即状态不可变，也无需同步措施。

让外部无法修改成员变量。成员变量加上final关键字。不提供修改变量或返回变量引用的方法。

volatile、CAS操作和加锁

保证类的可见性。适合一个线程写，多个线程读的场景。

例如ConcurrentHashMap，将 V 设置为 volatile 类型。get操作不加锁，put操作使用CAS操作或加锁。

ThreadLocal

每个线程保存一份资源的副本，互相不影响，就不存在线程安全问题。

Servlet不是线程安全的。经典Web交互模型使用Thread-per-Request，一个请求对应一个服务器线程的处理方式。且大部分使用消费队列的架构模式，都会将产品的消费过程尽量在一个线程中消费完。

死锁

两个或两个以上的进程在执行过程中，由于竞争资源或者由于彼此通信而造成的一种阻塞的现象，若无外力作用，它们都将无法推进下去。

发生死锁的实质，是获取锁的顺序不一致。

解决死锁

使用System.identityHashCode() （不受用户重写HashCode方法影响），计算需要加锁对象的哈希值大小，使用固定的加锁顺序。如果哈希冲突，则在类中另外定义一个静态成员锁，让多个线程竞争这个锁。

在对象中定义锁，使用锁的tryLock()。

活锁

如果使用tryLock的方式解决死锁。多个线程在竞争锁的时候，由于无法一次性获取所有的锁，一直重复尝试-失败-尝试-失败的过程。

可以让线程在获取锁失败后休眠一个随机时长，将不同线程竞争锁的时机错开。

线程饥饿

低优先级的线程，总是无法得到执行。

性能和思考

使用并发的目标是为了提高性能，引入多线程之后，相对单线程是有额外开销的。

  1. 一般原则，先保证程序正确，再提高速度。
  2. 性能表现以测试为准。

影响性能的因素

  • 上下文切换。5000~10000个时钟周期。
  • 内存同步。加锁。
  • 阻塞。线程的挂起。包括两次额外的上下文切换。

可以提升的角度

减少锁的竞争

  • 减少锁的粒度。使用锁时，其保护的对象是多个，然其中只有部分对象会被访问修改，则可以换成多个锁独立保护各个对象。需要注意避免发生死锁。
  • 缩小锁的范围。尽量缩短持有锁的时间。另外需要避免多余的缩减缩范围的操作。如果两次获取锁的时间间隔太短的话，虚拟机可能会自动执行锁粗化。
  • 锁分段。ConcurrentHashMap即使用的分段锁。
  • 替换独占锁。
    • 使用读写锁
    • CAS操作

线程安全的单例模式

懒汉式-双重检查

现在一般不建议使用了。

这种方式也并不是完全安全的。问题在于实例化函数 new Singleton()。并不能保证线程安全。

singleton = new Singleton(); 这段代码其实是分为三步执行：

  1. 为singleton分配内存空间
  2. 初始化singleton
  3. 将singleton指向分配的内存地址

但是由于JVM具有指令重排的特性，执行顺序有可能变成 1->3->2 。此时线程1已经执行了1和3步，线程2若调用getInstance，判断 null == singleton 时会返回true，从而拿到了未完成初始化的singleton实例。

可以将singleton变量设置成 volatile ，禁止JVM重排序代码。

饿汉式

在类初始化时就将实例创建好，由虚拟机来保证线程安全。

懒汉式-类初始化模式

也称延迟占位模式。此模式较为常用。
