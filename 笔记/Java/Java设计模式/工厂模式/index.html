工厂模式

发表于 2022-06-01 分类于 设计模式

前言

场景：吃苹果

谁想吃苹果，需要自己种苹果树。

  • 使用者需要自己负责new Apple()
  • 使用者需要管理果树，即Apple()类
  • 使用者需要知晓Apple实例化时的各项参数含义，如Apple(String name, int price);

简单工厂模式

工厂模式的基础。

由园丁管理果园，使用者向果园（工厂）请求水果，得到返回的对象。

  1. 将对象的创建和使用分开
  2. 将生产过程集中后，便于集中管理（增删改）
  3. 当水果类再有变动时，使用者不需要再去修改代码了

使用者使用

当新增水果类型时，只需改造StaticFactory类的代码即可。使用者对于水果类的使用方式无需再做改变。

工厂方法模式

在简单工厂模式中，随着需要实例化的对象类型越来越多（果园生产的水果种类越来越多），工厂类会变得越来越难以管理，方法扩展其为庞大，水果品种的扩展方式不优雅。

简单工厂模式实际上
不符合单一职责原则，一个类负责了各类对象的创建。
不符合开闭原则，扩展品类时，需要修改已有代码。

  • 为了弥补这种扩展性，在工厂方法模式中，将工厂方法抽象出来，变成单独的接口。
  • 工厂类结构与产品类结构一一对应，每一种产品都对应一个工厂子类。
  • 当新增一个产品类型时，新加对应的工厂子类即可，不需要修改既有类。

跟简单静态工厂模式相比，工厂类膨胀太多，需要根据需求取舍。

对于之前的例子来说，就是

  • 将静态工厂打散，每一个水果品类，对应一个水果品类工厂来生产。
  • 当需要扩展张水果品类时，对应扩展一个水果品类的工厂。

实现具体工厂

使用者使用

抽象工厂模式

当一个类别的产品还有多个系列区分时，为了按系列生产产品，使用抽象工厂区分。

当有多个工厂，且工厂之间有对应关系时，此对应关系也需要维护。

例如水果工厂类，和水果包装的工厂类，需要一一对应。如果人工维护可能出错，苹果用了香蕉的包装。

为了解决上述问题，可使用抽象工厂模式，将多个工厂合并成一个。

如此，使用者只需要实例化一个AppleFactory就可以完整正确的包装了。

建造者模式

建造者模式本质上也是一种工厂。

水果店推出水果套餐。需要为套餐建立对象，例如节假日套餐，会员套餐等。为各水果类设置价格并设置折扣。

需要建造的对象较为复杂，但步骤也比较固定，都是 客户下订单-设置苹果加个-设置香蕉价格-设置橘子价格-设置折扣-结算。可以使用建造者模式。

创建一个具体实现类

导演类 实例化 建造者，建造者 接收信息 装配对象 并返回，导演类拿到对象使用。

总结

原则：

  1. 解耦。将对象的创建和使用分开。
  2. 将创建过程黑盒化。工厂负责对象的创建，包括其初始化方法的调用。
  3. 面向接口编程。使用者只需知其接口，不需知其实现类。

对比：

  • 静态工厂模式。将所有对象的创建逻辑集中到一个类中处理。
  • 工厂方法模式。一个工厂负责创建一个产品类的创建。
  • 抽象工厂模式。将一个系列产品的工厂合并成一个工厂，负责生产这个系列的产品。
  • 建造者模式。对象的创建比较复杂时，按照步骤逐个创建对象进行装配，并让创建过程模板化。
