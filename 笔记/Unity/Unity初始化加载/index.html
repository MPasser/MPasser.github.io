Unity初始化加载

发表于 2026-01-26 分类于 Unity

前言

很多游戏启动后都会先进入一个加载页，用于完成初始化逻辑，作用类似计算机的 Boot 流程。

在 Unity（尤其是移动端）里，常见做法是：先进入一个独立的初始化场景，完成资源与系统初始化后，再切换到真正的游玩场景。

这类初始化场景通常只包含一套简单 UI（进度条 + 当前任务文案），流程相对固定，因此适合抽象为可复用方案。

方案设计

加载任务

将在初始化场景进行的加载任务分为三部分：加载主场景、自定义加载任务和等待最小加载时间。

自定义任务本身是串行执行的；“并行”的是主场景异步加载与任务执行过程。这两部分任务都完成后，再去执行等待最小加载时间的逻辑。

进度信息

总进度由两部分组成：

  • 场景加载进度（权重默认 60%）
  • 任务进度（权重默认 40%，且内部按任务 Weight 分配）

核心计算逻辑：

加载主场景和自定义加载任务并行执行，总进度为二者各自的进度与权重相乘后相加。

加载主场景的异步操作，其allowSceneActivation被设置为false，所以最大进度是0.9f（max_load_scene_progress），因此在计算其进度时，要用实际进度除以此值。

UI更新

BootController 将显示进度拆成两层：

  • actualProgress：逻辑上的真实进度
  • currentProgress：UI 当前显示进度

在 Update() 中通过 maxProgressSpeed 让 currentProgress 逐步追赶 actualProgress，形成平滑动画，避免进度条突跳。

SetProgressBar() 和 UpdateCurrentTaskName() 保持空实现，用于接入自定义 UI 方案。

自定义任务

设计一个接口，用于封装自定义加载任务

这个接口让启动逻辑具备很好的可扩展性，
只需新增一个实现类并在 RegisterBootTasks() 中注册即可。

固定的预定义任务

WaitMinBootDura是固定的最后一个任务。目的是防止加载速度过快，加载页显示时间太短。设置了一个最小时长，当所有任务执行完毕后，如果未达到此时长，则等待直至达到这个时长。

WaitMainSceneLoadComplete任务是固定的倒数第二个任务。目的是当自定义任务都执行完后，等待主场景加载完毕。这个任务的权重是0，因为加载主场景的进度值会在外部计算。

过渡动画

有些需求可能需要一个初始化场景和游玩场景之间的过渡动画。由于切换场景时中间有帧空隙，所以仅仅让初始化场景的最后一帧和游玩场景的第一帧相同无法达到很好的效果。

最简单的方案是使用常驻对象DontDestroyOnLoad，过渡动画播放完后销毁自身。
